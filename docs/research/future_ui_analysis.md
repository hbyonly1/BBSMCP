# BBS-MCP 创作控制台（UI/UX）系统设计分析

本文档记录了关于是否为 BBS-MCP 设计独立前后端管理系统的分析与建议，供未来开发参考。

---

## 1. 核心需求与痛点分析

在当前的基于 MCP 的对话式动画创作流程中，存在以下核心痛点：

| 问题描述 | 现状痛苦程度 | UI 是否能解决 |
| :--- | :---: | :--- |
| **资产不可见性** | 无法直观查看所有 Anchor 及其勘探截图 | ✅ 强烈推荐 |
| **蓝图盲盒** | 建筑 JSON 设计后，必须在游戏内加载才能预览 | ✅ 显著提升 |
| **状态持久化** | 剧本、分镜和已生成的资产在对话记录中，难以管理 | ✅ 场景管理 |
| **环境上下文** | 每次开启新对话时，AI 需要重新理解现有的场景布局 | ✅ 数据共享 |

## 2. 系统定位与结论

**结论：有必要设计，但建议分阶段实施。**

BBS-MCP 目前处于**创作管道（Creative Pipeline）的底层基础设施**阶段。过早投入重度前后端开发可能导致设计方案与实际工作流不匹配。

- **短期建议**：维持当前的 MCP 对话驱动模式，通过 AI 自动生成的 Artifacts（如分镜单）作为临时的可视化中转。
- **中期建议**：构建一个轻量级的“资产看板（Asset Dashboard）”，重点解决可视化问题。

## 3. 推荐路径：最小有价值 UI (MVP)

建议第一个 UI 版本不包含复杂的编辑功能，而是一个**只读且支持 3D 预览的资产浏览器**。

### 核心功能模块：
1. **Anchor 资产库**：
   - 展示缩略图墙。
   - 快速查看每个锚点的语义描述和相关的机位建议（Camera Hints）。
2. **建筑预览器（核心价值）**：
   - 提供一个基于 WebGL（如 Three.js）的简易方块预览器。
   - 允许在不启动 Minecraft 的情况下，直接查看 AI 生成的建筑 JSON 的外观和空间结构。
3. **分镜卡片流**：
   - 按照时间轴展示分镜（Storyboard）图片。
   - 对应每个镜头的文字描述、演员位置和摄像机运镜数据。

## 4. 衡量标准：工具 vs 创作

在决定进一步开发前，请始终权衡：**你愿意花多少时间在构建工具本身上，而不是用它来制作动画？**

> [!TIP]
> **最佳实践建议：**
> 先使用现有的 MCP + 对话工作流完整制作 1-2 个短片。通过实际的生产过程，识别出阻碍创作效率的真实“死穴”在哪里（例如：到底是改坐标累，还是选材质累？），然后再针对性地设计 UI 功能。
